1Core Authorization Concepts

Role Types

  • Single Role — one PFCG role, one set of menu + auth data.
  • Composite Role — a container of single roles; has no auth data of its own, just bundles roles for easier assignment.
  • Derived Role — inherits menu & authorizations from a master role; only org level values (e.g. Company Code, Plant) differ per derived role.
  • Enabler / Parent-Child pattern used heavily for plant / company-code-specific access without rebuilding menus.

PFCG Build Flow

Menu Tab → Authorizations Tab → Maintain Org Levels → Generate Profile → Add Role to User → User Comparison
  • Menu tab: add T-Codes / Fiori catalogs & groups / web links.
  • Authorizations tab: values pulled from SU24; yellow = needs org value, red = needs field value, green = complete.
  • Expert Mode: used to manually add/edit auth objects not proposed by SU24.
  • Generate creates the authorization profile (Z-profile) tied to the role.

Authorization Object Structure

Object Class → Authorization Object → up to 10 Fields → Field Values

S_TCODE (object) → TCD (field) → MM03 (value)

3Role Build, Transport & Maintenance

Naming Convention (typical)

PrefixMeaning
Z_S_...Single role
Z_C_...Composite role
Z_D_...Derived role
Z_FF_...Firefighter ID role (GRC EAM)

Derived Role Logic

Master role defines menu + non-org authorizations. Each derived role copies that data and only org level values (Company Code, Plant, Sales Org, etc.) are re-entered per derived role. Menu changes on the master automatically flow down.

Changing a field that is not an org level on the master and forgetting to re-generate derived roles is a classic audit finding — always regenerate + user-compare after master changes.

Role Transport Process

PFCG (Dev) → Save/Generate → Transport Request → STMS Release → Import to QAS → Import to PRD
  • Role transports carry both the menu (agr_*) and the generated profile.
  • Org values are not transported automatically in some setups — always confirm and re-generate/re-derive in target client if org values were changed locally.

User Comparison

T-CodeUse
PFUDMass user comparison (background job — run regularly)
PFCG → User tabIndividual role → user comparison
SUPCMass role generation / profile regeneration

Comparison pushes the generated profile into the user master (USR04) — without it, a newly built/changed role has no effect even if assigned.

SoD-Aware Role Design

  • Keep roles single-purpose / task-based rather than broad "do everything" roles.
  • Avoid combining conflicting business functions (e.g. Create Vendor + Post Payment) in one role.
  • Run GRC risk analysis on role design before go-live, not just at user assignment.
  • Use Firefighter (EAM) for rare elevated access instead of building it into standard roles.

6GRC Access Control — Overview

GRC Access Control (AC 12.0 / GRC on S/4HANA) governs who can request, get, and use access across connected SAP & non-SAP systems, layered on top of standard PFCG security. Four pillars:

ARA
Access Risk Analysis — finds & mitigates Segregation of Duties (SoD) risk in users/roles
ARM
Access Request Management — workflow-driven access provisioning
EAM
Emergency Access Management — controlled, logged "Firefighter" elevated access
BRM
Business Role Management — standardized methodology for building/maintaining roles

High-Level Architecture

GRC System (NWBC / Fiori Launchpad) ⇄ Connectors (RFC / Web Service) ⇄ Backend Plug-in (/GRCPI/...) ⇄ ECC / S4HANA / Non-SAP
  • Backend systems run the GRC plug-in so GRC can read roles, users, and authorization data and push provisioning requests back.
  • BRF+ (Business Rule Framework plus) drives dynamic decision logic — used heavily in ARM routing and MSMP workflow initiator rules.
  • MSMP (Multi-Stage Multi-Path) is the workflow engine behind Access Requests and other GRC processes.

Global / Config Transactions

T-CodeUse
NWBCClassic GRC launchpad (Fiori apps used in newer versions)
SPROIMG — GRC Access Control configuration node
SM30Maintain GRC config tables (e.g. connectors, rule sets)
GRFNMW_DEV_RULE / GRFNMW_CONFIGURE_WDMSMP workflow configuration & rule assignment
BRF+ (Workbench)Build initiator / routing / agent rules for workflow
GRAC_SPMEmergency Access (Firefighter) launch & log review
Interview one-liner: "GRC AC doesn't replace PFCG security — it sits on top of it to govern the request, risk, emergency-access, and role-lifecycle processes around that same authorization model."

7ARAAccess Risk Analysis

Key Terms

TermMeaning
Risk IDA defined SoD or critical-access risk, e.g. "Create Vendor + Post Payment"
FunctionGroups related Actions/Permissions that represent one business task
ActionA T-Code / Fiori app tied to a Function
PermissionThe authorization object/field/value behind an Action
Rule SetThe full library of risks used for analysis (SAP-delivered "Global Rule Set" or a custom one)
Mitigating ControlA compensating control (e.g. a monthly report review) assigned to accept a risk

Risk Types

  • SoD Risk — conflict between two or more Functions (e.g. Create PO + Approve PO)
  • Critical Action — a single sensitive T-Code/app (e.g. SU01, PFCG)
  • Critical Permission — a single sensitive authorization value

Risk Levels

LevelTypical meaning
CriticalHigh-impact fraud/financial risk — must be mitigated or remediated
HighSignificant risk — needs mitigation or role redesign
MediumModerate — monitor / mitigate depending on org policy
LowMinor — usually accepted or lightly monitored

Analysis Types

  • User-level analysis — what SoD conflicts a specific user actually has (across assigned roles)
  • Role-level analysis — SoD risk baked into a role's design, before it's ever assigned
  • Simulation — "what-if" analysis before adding a role/access to a user, to catch new conflicts pre-emptively
  • Batch Risk Analysis — scheduled background job to keep risk data current org-wide

Mitigation Lifecycle

Risk Identified → Mitigating Control defined → Control assigned to User/Role → Control Monitor performs periodic review → Recertification / re-approval

A Mitigating Control has a Control Owner (defines it), a Monitor (performs the review), and an Approver (approves the assignment) — segregating the mitigation process itself.

8ARMAccess Request Management

Request Types

  • New Account — first-time system access
  • Change / Modify — add/remove roles for an existing user
  • Extend Validity — extend an expiring role assignment
  • Lock / Delete — deactivate or remove access (e.g. leaver process)
  • Firefighter ID Assignment — request temporary elevated access (feeds EAM)
  • Mass Access Request — bulk requests, e.g. new-hire batches

Workflow Engine — MSMP

Request Created → BRF+ Initiator Rule picks the correct Path/Process → Stage 1 (e.g. Manager Approval) → Stage 2 (Role Owner) → Stage 3 (Security Approval) → Provisioning
  • Process ID — the overall workflow definition (e.g. Access Request process)
  • Path — a specific route through stages, chosen dynamically by rule (e.g. by request type, risk level, org unit)
  • Stage / Agent — each approval step and who (role or named approver) can act on it

Embedded Risk Analysis

SoD risk analysis runs automatically during the request (not just after). If the new access creates a conflict, the requestor/approver sees the violation and must:

  • Mitigate — assign an existing mitigating control
  • Remediate — remove the conflicting access from the request
  • Accept & proceed — if policy allows, with justification/approval

Provisioning Types

TypeBehavior
Automatic / DirectGRC pushes the role/access straight into the backend system via connector
IndirectAccess is granted by assigning a business role that maps to backend roles
No ProvisioningManual/offline — GRC just documents the approval, security team applies it manually
Interview line: "ARM isn't just an approval chain — it embeds SoD risk analysis at request time, so violations get caught before provisioning, not after."

9EAMEmergency Access &BRMBusiness Role Mgmt

EAM — Key Roles

RoleResponsibility
Firefighter IDSpecial user ID pre-loaded with elevated/emergency access
Firefighter (User)The person who logs in "as" the Firefighter ID for a task
Firefighter OwnerApproves which users can use which Firefighter ID
Firefighter ControllerReviews the activity/change log after each use

EAM Flow

User launches Firefighter ID (GRAC_SPM / Fiori app) → enters Reason Code + description → performs work under FF ID → session logged (actions, tables, changes) → Controller reviews log → sign-off / escalation if needed
  • Centralized Firefighting — access launched and controlled from the GRC system itself.
  • Decentralized Firefighting — the Firefighter ID is assigned directly in the backend, GRC captures logs after the fact.
  • Reason codes are mandatory — they document why emergency access was needed, for audit.

BRM — Core Idea

BRM standardizes how roles are designed, built, and maintained across the landscape, rather than each consultant building roles ad-hoc in PFCG.

  • Business Role — a role defined from a business/process perspective (e.g. "AP Clerk").
  • Technical Role — the underlying PFCG role(s) in one or more backend systems that a Business Role maps to.
  • Role Mining — analyzes existing usage/access patterns to suggest role content instead of designing from scratch.
  • Role Owner / Approver — accountable for what the role contains and who gets it.
  • Role Attributes — metadata (org level, functional area, risk level) used to classify and search roles.
BRM enforces a consistent role lifecycle: Define → Build/Derive → Test → Approve → Deploy → Periodic Review — with SoD risk checked at each stage.

10Top Interview Q&A + Quick Revision

Difference between a Composite and a Derived role?
Composite just bundles single roles for assignment convenience and has no auth data itself. Derived inherits menu + non-org authorizations from a master role and only needs org values maintained separately.
What is SU24 and why does it matter?
It stores the proposed authorization objects/values per T-Code (check indicators). PFCG pulls from SU24 when building a role — if SU24 is wrong or unmaintained, the role build will be incomplete or inconsistent.
User reports "no authorization" — first three things you check?
Reproduce the error, run SU53 immediately, then check the owning role in PFCG (SUIM: Roles by User) for the missing object/value.
How do you handle a SoD conflict found during an access request?
Either mitigate it with an existing mitigating control, remediate by removing the conflicting access, or get formal risk acceptance/approval per company policy — never just push it through.
What's the point of Firefighter IDs if we already have roles?
They provide time-boxed, logged, reason-coded elevated access for exceptions (e.g. period-end, incident fixes) instead of permanently widening a user's standard role — keeping day-to-day access lean.
Difference between IWSG and IWSV?
IWSG is the frontend Gateway service that starts a Fiori app; IWSV is the backend service supplying the real authorization objects. Missing IWSG = app won't launch; missing IWSV = app opens but data/actions fail.
What's the difference between Access Risk Analysis and Access Request Management?
ARA is the risk engine — it finds and helps mitigate SoD/critical-access risk. ARM is the provisioning workflow — it routes, approves, and provisions access requests, calling ARA's risk engine as part of that process.
